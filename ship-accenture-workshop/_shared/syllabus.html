Presentation philosophy
  • Demo often. 3–5 live demos, every time — even a 1-hour 101. Sprinkled throughout, not bolted on at the end. The audience's belief tracks what they see working, not what you assert on a slide.
  • Tailor to the customer. A bug in portfolio-service lands with finance engineers in a way the generic feature-flag service never will. Domain alignment isn't cosmetic — it's the credibility beat. /ncp generates a vehicle in their domain for exactly this reason.
  • Same arc, customer-tuned. Every deck follows the same 101 → 201 → 301 spine. What changes is the demo vehicle, the planted bug, the CLAUDE.md, and the cobrand. Consistency across the field; relevance in the room.
  • Patterns, not rules. There is no one right way to use Claude Code. CLAUDE.md, plan mode, skills — these are patterns. Tell the room to steal what's useful and ignore what isn't.
  • Show the failure first. Open with the failing test (Cold Open, Demo 1). Watching Claude find and fix a real bug in 90 seconds is the hook — everything after is "here's how to make that repeatable for you."
  • Match the room. New-to-CC audiences get 101 + light 201. Power users skip straight to 201 + 301. Mixed rooms get the Intro track with one 301 teaser (usually Agent Teams). /ncp asks for audience and picks accordingly.
  • Smoke-test before you present. npm test shows exactly 1 failure, terminal font ≥ 18pt, notifications off, a warmed-up claude session in a second tab. Preflight is in every deck's demo-scripts.html.
Time budget: ~40% slides, ~35% live demos, ~25% hands-on exercises (when length allows). If you're cutting for time, cut slides — never demos.
Track A
Intro — 101 → 201
Audience: new to Claude Code, or mixed · 60–90 min · 3–4 demos, 1–2 exercises
Walk-out outcome

Everyone has installed Claude Code, fixed a real bug in the demo repo, written a CLAUDE.md, and run plan mode. They understand the agent loop and know which model to pick.

101 — Foundations

  • What is Claude Code — agent in your terminal; reads, edits, runs
  • The agent loop — Read → Decide → Act → Observe
  • Install & auth — one curl, platform-switched
  • Models — Opus / Sonnet / Haiku; which model when
  • The core loop — prompt → read → plan → edit → review → loop
  • Feature ladder — 7-rung maturity, chat → agent teams
  • Where it runs — CLI, IDE, Web, everywhere
  • How Anthropic uses it — credibility beat; cross-org value beyond engineering
  • No one right way — patterns not rules
  • Rookie moves — 6 things even experienced users get wrong
  • Demo 1 — Fix the test — cold loop; find and fix the failing test
  • Exercise 1 — Reproduce & fix — hands-on; clone, find the bug, fix it

201 — Teach Claude your repo

  • Context is everything — output quality = context quality
  • CLAUDE.md — the README for the AI; under 200 lines; nesting across home/repo/subdir
  • Plan mode — Shift-Tab twice; read-only, propose, zero edits until approved
  • Input primitives — ! bash mode, @ file mentions, screenshots, slash commands
  • Spec-driven & TDD — define-success-first; red → green → refactor
  • Skills — domain expertise, codified; markdown → slash command; SKILL.md anatomy
  • Context management — /context, /compact, /resume, auto-memory
  • Effort levels — dial reasoning depth; adaptive thinking
  • Prompting best practices — be specific, give context, iterate
  • Demo 2 — Plan mode — read-only planning, refine, no edits yet
  • Demo 3 — Run a skill — custom slash command fires, then reveal SKILL.md
  • Exercise 2 — Teach the repo — fill CLAUDE.md TODOs + write your first skill
If short on time: drop SDLC Carousel, Workflow Transformation, and the TDD deep-dive. Keep every demo.
Track B
Advanced — 201 → 301
Audience: already using Claude Code daily · 90–180 min · 4–5 demos, 2–3 exercises
Walk-out outcome

Everyone has wired an MCP server, run subagents in parallel, packaged a skill into a plugin, and seen agent teams split work across a tmux session. They know the enterprise/CI surface area.

201 — Teach Claude your repo (compressed — assume install is done)

  • CLAUDE.md & nesting — quick recap; focus on what shouldn't go in it
  • Plan mode — as a refactor-safety tool, not just onboarding
  • Skills & SKILL.md anatomy — frontmatter, body, bundled assets
  • Hooks — pre/post tool events; block unsafe edits; full lifecycle catalog
  • MCP architecture — client/server; tools, resources, prompts over stdio/SSE/HTTP
  • Checkpointing & /rewind — three restore modes; code + conversation undo
  • Effort levels & adaptive thinking — when to override with ultrathink
  • Demo 3 — Run a skill — then reveal SKILL.md
  • Demo 4 — Hooks block edits — pre-edit hook blocks unsafe path; Claude routes around

301 — Scale & integrate

  • Parallel patterns — why parallelize; sequential vs parallel; three patterns
  • Subagents — Explore / Plan / General; isolated context
  • Custom agents — define your own in .claude/agents/ — model, tools, isolation
  • MCP — USB for tools; claude mcp add; managed MCP for org-wide connectors
  • Plugins — bundle skills + MCP + hooks; one install, whole team inherits; must-have starter set
  • Agent teams — tmux split; many Claudes, one repo, filesystem as message bus
  • Parallel compare — subagents vs parallel Claude vs agent teams
  • Surfaces beyond the CLI — Claude Code Web, scheduled tasks, mobile approve, remote control
  • Enterprise & CI — policy deployment, runtime hardening, headless --bare -p
  • Demo 5 — Parallel Explore — 3 Explore agents fan out, results merge back
  • Demo 6 — Wire up MCP — add server, ask a question, tool call appears
  • Demo 7 — Agent teams — tmux split, three Claudes, one feature
  • Exercise 3 — Wire up MCP — hands-on
  • Exercise 4 — Package it — bundle your skill into a plugin, validate green
If short on time: drop Mobile, Remote Control, and the Parallel Compare matrix. Keep Demo 7 — agent teams is the slide people photograph.
